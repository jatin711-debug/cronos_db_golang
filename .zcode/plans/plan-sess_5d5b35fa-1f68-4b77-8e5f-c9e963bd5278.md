Production Hardening Plan for CronosDB — approved by user.

DECISIONS CONFIRMED:
1. WAL on-disk format change for per-entry terms is acceptable; clean data directory required (no v1->v2 migration).
2. Add explicit `--dev` insecure mode; production mode requires TLS, auth, encryption, RF>=3, minISR>=2.
3. Implement all phases in one continuous push on the `dev` branch.

PHASES:
- Phase 0: Fix dedup deadlock, idempotent shutdown (sync.Once + WaitGroup), CI health-check port, README claims, partition metadata from WAL.
- Phase 1: Extract atomicWriteFile helper, add fsync to follower bulk sync, offset store, DLQ, audit, backup, and make tx coordinator fail on tx log errors.
- Phase 2: WAL per-entry terms + segment format v2 with v1 fallback, replication AppendEntries checksums, fix leader NextOffset bookkeeping, protect transport map.
- Phase 3: Persist consumer group metadata/assignments and exactly-once commit IDs to PebbleDB.
- Phase 4: Rewrite retention (index timestamps, high-watermark guard), CDC bounded worker pool, security defaults & `--dev` mode, gRPC graceful drain, default fsync mode `batch`.
- Phase 5: Chaos tests, fix gRPC durable-write benchmark, re-run WAL/gRPC/loadtest benchmarks, update docs.

THROUGHPUT PROTECTION:
- New fsyncs are batched, periodic, or off the hot path.
- WAL term adds 8 bytes/record (~0.2% for 4KB payloads).
- Replication checksum is per-batch CRC32.
- Consumer metadata async/batched.
- CDC worker pool bounded.

Each phase will compile, pass tests, and throughput-sensitive changes will be benchstat-validated where applicable. Commit/push to `dev` when complete.
